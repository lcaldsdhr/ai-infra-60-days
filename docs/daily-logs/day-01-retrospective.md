## 一、3 句话总结（最重要的认知）

1. **Transformer 三大基础件**：Q/K/V 三剑客，Q 查 K 索引 V 内容，√d_k 缩放保 softmax 数值稳定；Mask 分 Causal（防未来）和 Padding（防 pad）两类，训练-推理必须一致；RoPE 用旋转矩阵编码位置，让内积天然只依赖 (m-n) 相对距离，强外推性。

2. **大模型训练的 4 大瓶颈**：显存（12 字节/参数）、算力（6N FLOPs）、通信（10-15% 损耗）、数据（流式加载），对应 AMP+FSDP / TP+PP / overlap+压缩 / DDP 四套方案。

3. **为什么必须分布式**：单卡 H100 80GB 装不下 70B 训练（需 840GB），单卡算 70B@1T 要 500 天；分布式提供 8 卡 ≈ 8× 显存 + 7-8× 算力，但付出通信、同步、复杂度三重代价。

## 三、概念对照表（闭卷默写版）

### 表 1：训练 vs 推理

| 维度 | 训练 | 推理 |
|---|---|---|
| **计算图** | 完整（前向 + 反向 + 优化器） | 仅前向 |
| **显存组成** | 参数 + 梯度 + 优化器状态 + 激活 + 临时 | 参数 + KV cache（生成式） |
| **显存大小** | 大（≈ 12-16 字节/参数） | 小（≈ 2 字节/参数 + KV） |
| **算力强度** | 大（70B 模型 ~10²³ FLOPs） | 小（每个 token ~10¹⁰ FLOPs） |
| **耗时占比** | 反向 ≈ 2× 前向 | 仅前向 |
| **通信** | DP all-reduce / TP all-reduce / PP send-recv | 几乎无（除 TP） |
| **容错** | 低（一个 NaN 全部废） | 高（单请求失败可重试） |
| **优化目标** | 吞吐（tokens/sec/GPU） | 延迟（TTFT / TPOT） |
| **batch 大小** | 大（梯度稳定） | 1~几十（受延迟约束） |
| **典型框架** | Megatron / DeepSpeed / FSDP | vLLM / TensorRT-LLM / SGLang |

**关键洞察**：训练是 **throughput-bound**（要算得快、并行多），推理是 **latency-bound**（要响应快、首 token 短）。

### 表 2：数据并行 vs 模型并行

| 维度 | 数据并行 (DP / DDP) | 模型并行 (TP / PP) |
|---|---|---|
| **切分对象** | 样本（按 batch 切） | 模型权重 / 激活（按层切） |
| **每卡显存** | 完整模型 | 切片的模型 |
| **通信频率** | 每 step 1 次 all-reduce | 每层 1 次（TP）/ 每 stage 1 次（PP） |
| **通信量** | 中（与参数量成正比） | TP 大（all-reduce 每层）/ PP 小（点对点） |
| **扩展性瓶颈** | batch size 上限 | 模型规模上限 |
| **适合模型** | 中小（< 10B） | 大（> 10B） |
| **典型实现** | DDP / FSDP / ZeRO | Megatron-TP / PipelineParallel |
| **同步成本** | 高（all-reduce 同步等） | TP 中 / PP 更高（bubble） |

**变体**：
- **ZeRO-1/2/3**：把优化器状态/梯度/参数也分片到 DP 维度
- **FSDP**：PyTorch 原生 ZeRO-3 等价物
- **TP + PP 混合**：GPT-3 175B 标配（8 路 TP × 12 路 PP）

### 表 3：延迟 vs 吞吐

| 维度 | 延迟 (Latency) | 吞吐 (Throughput) |
|---|---|---|
| **定义** | 单请求耗时 | 单位时间处理量 |
| **单位** | ms（首 token / 每 token） | req/s 或 token/s |
| **核心指标** | TTFT (Time To First Token) | TPS (Tokens Per Second) |
| **优化方向** | 减关键路径 | 增 batch / 流水线 |
| **关键技术** | KV cache / Speculative Decoding / 量化 | Continuous Batching / FSDP / 大 batch |
| **场景** | 实时对话、搜索 | 离线批处理、内容审核 |
| **硬件利用** | 经常 < 50% | 经常 > 80% |
| **典型权衡** | 延迟优先 = 小 batch | 吞吐优先 = 大 batch |

**关键洞察**：
- **延迟敏感**：batching 难做大（用户等不了），靠**算法优化**（speculative、quantization）
- **吞吐敏感**：batching 可以很大，靠**硬件利用**（大 batch、流水线）

### 记忆口诀

> **训练求吞吐，推理求延迟；数据并行切样本，模型并行切权重；**
> **DP 每步同步一次，TP 每层同步一次，PP 每个 stage 同步一次。**

## 四、面试卡 1：训练大模型的核心难点 + 方案

### 总框架（4 难点 → 4 方案）

```
训练大模型的 4 大难点
  ├── 1. 显存不够  →  显存优化（分片 / 重计算 / 混合精度）
  ├── 2. 算力太慢  →  并行计算（TP / PP / SP）
  ├── 3. 通信瓶颈  →  通信优化（overlap / 压缩 / 拓扑）
  └── 4. 数据太大  →  数据并行（DDP / FSDP / 流式加载）
```

### 1. 显存不够 → 显存优化

**显存组成（5 大部分）**：

```
总显存 ≈ 参数 + 梯度 + 优化器状态 + 激活 + 临时 buffer
       ≈ 2B + 2B + 8B + (B·S·H) + ...
       ≈ 12 字节/参数（Adam + bf16）
```

**优化方案（从轻到重）**：

| 方案 | 节省 | 代价 |
|---|---|---|
| **混合精度（BF16）** | 权重 50% / 梯度 50% | 略损精度 |
| **Gradient Checkpointing** | 激活 60-80% | 多 30% 计算 |
| **CPU Offload** | 优化器状态 100% | 慢 5-10× |
| **ZeRO-1（优化器分片）** | 优化器 N× | 通信 +1 次 |
| **ZeRO-2（+梯度分片）** | + 梯度 N× | 通信 +1 次 |
| **ZeRO-3 / FSDP（+参数分片）** | + 参数 N× | 通信量大 |

**记忆口诀**：
> **"12 字节/参数"是基础，AMP 砍一半，Recompute 砍激活，FSDP 砍 12 全分**

### 2. 算力太慢 → 模型并行

**为什么单卡算力不够**：

```
H100 单卡：~1000 TFLOPS（BF16）
训练 70B 模型 1T tokens：~10²³ FLOPs
                    ÷ 1000 TFLOPS ÷ 86400s
                    ≈ 1000 卡·天
                    → 单卡 3 年
```

**3 类并行**：

| 并行方式 | 切分 | 通信 | 适合 |
|---|---|---|---|
| **TP（张量并行）** | 单层矩阵乘 | all-reduce 每层 | 大层 |
| **PP（流水并行）** | 层数 | send/recv 每 stage | 层数多 |
| **SP（序列并行）** | 序列维度 | all-gather + reduce-scatter | 长序列 |

**5D 混合并行**：TP × PP × SP × CP × EP 任意组合
- **GPT-3 175B**：8 TP × 12 PP = 96 卡
- **LLaMA 3 405B**：16 TP × 16 PP = 256 卡
- **Qwen3.5 397B**：FSDP × EP（MoE 专家并行 16 路）

### 3. 通信瓶颈 → 通信优化

**3 类通信问题**：

```
a) 通信量太大  →  通信量压缩（ZeRO++、FP16 gradient）
b) 通信拖慢计算 →  计算-通信 overlap（CUDA Graph、fused kernel）
c) 拓扑不匹配  →  拓扑感知（Ring / Tree / NVLink 优先）
```

**关键技术**：
- **NCCL**：英伟达集合通信库
- **Ring AllReduce**：每卡只跟邻居通信，O(N) 步
- **Fused Kernel**：把多个小算子合成一个，减少 kernel launch
- **CUDA Graph**：预先录制计算图，重复执行无 launch 开销

### 4. 数据太大 → 数据并行 + 流式

**单卡装不下数据时的方案**：

```
├── 数据并行（DDP）：N 卡各持一份模型，batch 切 N 份
│   ├── FSDP：参数也分片
│   └── ZeRO：分片粒度更细
├── 流式数据：IterableDataset + 异步预取
└── 数据格式：WebDataset / TFRecord / Parquet（避免小文件）
```

**关键数字**：
- DDP 通信量 = 2×参数量（每个 step）
- FSDP 通信量 ≈ DDP × 1.5（多了 reduce-scatter）

### 自检（5 分钟闭卷默写）

背诵模板：
> "4 个难点：显存 / 算力 / 通信 / 数据。显存用 AMP + Recompute + FSDP；算力用 TP/PP/SP；通信用 overlap + 压缩 + 拓扑；数据用 DDP + 流式。"

### 变形题（面试官爱追问）

| 追问 | 一句话答 |
|---|---|
| FSDP 和 DDP 区别？ | DDP 复制完整模型，FSDP 把参数也分片 |
| ZeRO-1/2/3 区别？ | 分别分片 优化器/梯度/参数 |
| TP 为什么每层都通信？ | 矩阵乘切完后，每头需要其他头的部分和 |
| PP 的 bubble 怎么消？ | 微批次（micro-batch）流水线填充 |

## 五、面试卡 2：为什么需要分布式训练？单卡极限

### 总框架（3 极限 → 3 收益 → 3 代价）

```
为什么需要分布式？
├── 单卡 3 个极限
│   ├── 显存极限（装不下模型）
│   ├── 算力极限（算得太久）
│   └── 扩展极限（无法变大）
├── 分布式 3 个收益
│   ├── 显存线性扩展
│   ├── 算力近线性扩展
│   └── 模型可上 100B+
└── 分布式 3 个代价
    ├── 通信开销
    ├── 同步开销
    └── 复杂度开销
```

### 一、单卡 3 个极限

#### 极限 1：显存装不下模型

```
H100 单卡 = 80 GB
训练 70B 模型：
  权重 BF16    = 70B × 2B = 140 GB  ❌ 装不下
  + 梯度       = 140 GB
  + 优化器     = 560 GB
  合计         ≈ 840 GB（10 倍不够）

→ 单卡连推理都勉强，更别说训练
```

**对比推理**：70B BF16 = 140 GB → 需要 2 张 H100（80GB×2）才能跑

#### 极限 2：算力算得太久

```
H100 单卡：~1000 TFLOPS（BF16）
训练 70B 模型在 1T tokens 上：
  总 FLOPs ≈ 6 × 70B × 1T = 4.2 × 10²² FLOPs
  单卡时间 = 4.2×10²² / 10¹⁵ = 4.2×10⁷ 秒 ≈ 500 天 ❌
```

**实际**：GPT-4 训练量约 2×10²⁵ FLOPs，单卡要 6 万年。

#### 极限 3：扩展性极限

```
单卡 = 0 通信，0 扩展
即使算力 / 显存翻倍，1 张卡 1 个任务，并行度 = 1
→ 永远只能处理"1 个 batch"大小的数据
```

### 二、分布式 3 个收益

| 收益 | 数字 | 解释 |
|---|---|---|
| **显存线性扩展** | 8 卡 ≈ 8× 显存 | 每卡装 1/8 模型（FSDP） |
| **算力近线性扩展** | 8 卡 ≈ 7-8× 速度 | 通信损耗 10-15% |
| **模型可扩展** | 100B+ 参数 | 单机装不下，分布式装得下 |

**算力扩展公式**（考虑通信）：

```
加速比 = N / (1 + (N-1) × α)
其中 α = 通信时间 / 计算时间

α = 0.05（通信占 5%）→ 8 卡加速 6.5×
α = 0.20（通信占 20%）→ 8 卡加速 3.3×
```

### 三、分布式 3 个代价

| 代价 | 表现 | 缓解 |
|---|---|---|
| **通信开销** | all-reduce / all-to-all 拖慢 | overlap、压缩 |
| **同步开销** | 慢卡拖整队（straggler） | 容错、弹性调度 |
| **复杂度** | 5D 并行编排、检查点、容错 | 框架封装（Megatron/DeepSpeed） |

### 自检（3 分钟闭卷默写）

背诵模板：
> "单卡 3 个极限：显存装不下、算力算太久、扩展不了。分布式 3 个收益：显存线性、算力近线性、模型可上 100B。3 个代价：通信、同步、复杂度。"

### 变形题（高频追问）

| 追问 | 一句话答 |
|---|---|
| 8 卡理论加速 8×，实际为什么只有 6-7×？ | 通信损耗 10-15% |
| 为什么不直接用 1 张大显存卡？ | 单卡 HBM 带宽、容量、价格不可线性扩展 |
| 100B 模型必须分布式吗？ | 训练必须；推理可用 CPU 卸载（慢）或量化（损精度） |
| 为什么 MoE 仍要分布式？ | 17B 激活 ≠ 17B 存储，权重仍 397B 要分片 |

### 关键数字汇总（脱口而出）

| 数字 | 含义 |
|---|---|
| **12 字节/参数** | Adam + BF16 训练每参数显存 |
| **6N** | 训练 transformer 的 FLOPs 估算（6 × 参数量 × tokens） |
| **80 GB** | H100 单卡显存 |
| **1000 TFLOPS** | H100 BF16 算力 |
| **10-15%** | 8 卡分布式通信损耗 |
| **N× → (N-1)× + 1** | 加速比公式近似 |
